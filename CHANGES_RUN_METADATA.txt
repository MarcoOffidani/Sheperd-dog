parent_run_id
- Fresh runs write an empty parent_run_id.
- Resume runs keep their own new run_id and write parent_run_id when it can be inferred.
- The parent is detected from --load-model-path by first looking for a run_id in the path, then by matching the path against experiment_name values already present in runs.csv.
- If no parent can be determined, parent_run_id is left empty for backward compatibility.

Lineage tracking
- runs.csv can be treated as a directed graph.
- run_id identifies the current run.
- parent_run_id points to the run that produced the loaded model or checkpoint.
- This keeps resumed runs traceable without merging their time-series rows under the original run_id.

- EnvStep
- global_timestep

- episode_idx
- TotalPedestrians
- padded

Why static metadata lives in runs.csv
- Static run parameters are written once per run_id in runs.csv.
- This keeps runs.csv as the source of truth for run configuration and avoids repeating long or mostly constant values on every time-series row.

Why escaped_counts.csv stays narrow
- escaped_counts.csv contains only per-step evacuation counts plus run_id.
- Join escaped_counts.csv to runs.csv on run_id to recover experiment_name and all static parameters.
- Buffering and flushing are unchanged; escaped_counts.csv still uses escaped_counts_buffer and flush_escaped_counts().

Compatibility notes
- Existing runs.csv files with the old header have fewer columns than the new schema.
- Existing escaped_counts.csv files from the previous run_id schema remain compatible with the new escaped_counts.csv schema.
- Older escaped_counts.csv files that used experiment_name instead of run_id need separate handling or regeneration.
